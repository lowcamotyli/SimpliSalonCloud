## 1. Co to jest v2.0

v1.0 to **operating system dla salonu beauty** — booking, CRM, billing, messaging, formularze. Działa w produkcji.

v2.0 to **treatment-capable salon OS** — ta sama platforma rozszerzona o pełen cykl życia zabiegu:

```
Rezerwacja
  → Formularz zdrowotny przed wizytą
  → Karta zabiegu (treatment record) z danymi protokołu
  → Plan wielosesyjny (jeśli potrzebny)
  → Dokumentacja fotograficzna
  → Audyt dostępu do danych zdrowotnych
  → Automatyczny follow-up (survey, kolejna sesja)
```

v1.0 miał kroki 1–2 i ostatni. v2.0 zamyka pętlę.

---

## 2. Target segment v2.0

### Główny: treatment-heavy beauty salon

Salony oferujące zabiegi wymagające dokumentacji, historii, protokołu:
- laserowe kliniki depilacji
- kliniki kosmetologiczne (kwasy, peelingi, mezoterapia)
- salony z urządzeniami hi-tech (HIFU, RF, kriolipoliza)
- medycyna estetyczna na poziomie light (bez lekarza w roli głównej)

### Warunki kwalifikacji segmentu
- Mają powtarzalnych klientów z historią zabiegów
- Wymagają zgód i wywiadów zdrowotnych
- Mają protokoły zabiegu (nie improwizują)
- Potrzebują dokumentacji do ubezpieczenia lub własnej ochrony prawnej
- Są skłonni zapłacić więcej niż podstawowy salon za funkcje kliniczne

### Poza targeten v2.0
- Aesthetic medicine / kliniki medyczne z lekarzem — to Stage 3 / v3.0
- Salony fryzjerskie / paznokcie — zostają na v1.0 feature secie
- Multi-location enterprise — osobny backlog, nie v2.0

## 3. Feature set v2.0

### 3.1 Wymagane prerekvizity (nie nowe, ale muszą być zamknięte)

| Feature | Sprint | Dlaczego bloker |
|---------|--------|-----------------|
| 3 migracje SQL (form_data_category, restrict_client_forms, health_consent_at) | TC-1 | GDPR compliance — bez tego karty zdrowotne nie spełniają Art. 9 |

### 3.2 Nowe domeny v2.0

#### Treatment Records (L2-A + L2-B)
Karta zabiegu tworzona po lub w trakcie wizyty przez pracownika.

**Musi zawierać:**
- powiązanie z bookingiem, klientem, pracownikiem, usługą
- parametry zabiegu (JSON — elastyczne per usługa)
- notatki zaszyfrowane AES-256-GCM
- data_category (general | health | sensitive_health)
- dostęp: owner/manager pełny, employee read-only własnych

**UI:** lista kart per klient, formularz tworzenia z widoku bookingu, widok pojedynczej karty

**Feature flag:** `treatment_records` (Professional+ plan)

---

#### Treatment Protocols (L2-C)
Wielokrotnie używane szablony parametrów zabiegu przypisane do usługi.

**Musi zawierać:**
- nazwa, opis, lista pól/parametrów z typami
- przypisanie do usługi (1:N)
- wersjonowanie (version: int, soft — nie event sourcing)
- przy tworzeniu karty: wybór protokołu → autofill pól

**UI:** settings/protocols — CRUD, assign to service

---

#### Multi-Session Treatment Plans (L2-D)
Plan N sesji zabiegu z śledzeniem postępu.

**Musi zawierać:**
- plan (nazwa, usługa, liczba sesji, protokół)
- sesje (numer, status: planned/completed/cancelled, powiązanie z bookingiem)
- postęp widoczny w profilu klienta
- automatyczny trigger: sesja zakończona → zaplanuj kolejną (hook w automation engine)

**UI:** widok planu w profilu klienta, progress tracker

---

#### Photo Documentation (L2-E)
Zdjęcia before/after powiązane z kartą zabiegu.

**Musi zawierać:**
- upload do Supabase Storage (`treatment-photos` bucket, private)
- consent gate: foto tylko gdy `health_consent_at` jest ustawione
- powiązanie z treatment record
- GDPR delete: endpoint do usunięcia zdjęć klienta
- thumbnail w widoku karty zabiegu

**Limit v2.0:** brak AI-comparisons, brak galerii publicznej — to v3.0

---

#### Clinical Audit Trail (L3-A — wymagane już w v2.0)
Immutable log każdego odczytu odszyfrowanych danych zdrowotnych.

**Dlaczego już w v2.0 (nie v3.0):** Kliniki treatment-heavy potrzebują tego do własnej ochrony prawnej, zanim wejdziemy w segment medyczny. Bez tego nie możemy sprzedać produktu poważnym kliniom.

**Musi zawierać:**
- log: kto, co, kiedy odczytał (health data access)
- append-only (brak UPDATE/DELETE przez RLS)
- viewer: owner-only w settings/audit-log
- Nie dotyczy: zwykłe bookings i CRM — tylko health data access

### 3.3 Poza zakresem v2.0 (świadome odroczenie)

| Feature | Dlaczego nie v2.0 | Kiedy |

## 4. Kryteria zamknięcia v2.0 (Definition of Done)

### Funkcjonalne (wszystkie must-pass)
- [ ] 120+ kart zabiegowych widocznych i importowalnych w `/settings/import`
- [ ] `conditionalShowIf` działa w publicznym rendererze formularzy
- [ ] Submit formularza zdrowotnego bez `health_consent_at` → HTTP 422
- [ ] Treatment Record można stworzyć z widoku bookingu przez owner/manager
- [ ] Treatment Record z notatkami zdrowotnymi szyfruje je AES-256-GCM przed zapisem
- [ ] Employee widzi karty zabiegu read-only, nie może tworzyć ani edytować
- [ ] Treatment Protocol można przypisać do usługi
- [ ] Formularz tworzenia karty zabiegu preloaduje pola z protokołu
- [ ] Treatment Plan z N sesjami jest widoczny w profilu klienta z progress trackerem
- [ ] Zdjęcia można uploadować do karty zabiegu (tylko gdy health_consent_at ustawione)
- [ ] GDPR: endpoint do usunięcia zdjęć klienta działa
- [ ] Audit trail loguje każdy odczyt odszyfrowanych danych zdrowotnych
- [ ] Owner widzi audit log w settings

### Techniczne (wszystkie must-pass)
- [ ] `npx tsc --noEmit` — 0 błędów
- [ ] `supabase db push` — wszystkie migracje zastosowane
- [ ] `supabase gen types typescript --linked` — typy zregenerowane po każdej migracji
- [ ] Żaden nowy `as any` dodany w domenach v2.0
- [ ] RLS policies na wszystkich nowych tabelach (treatment_records, treatment_protocols, treatment_plans, treatment_sessions, treatment_photos, health_data_access_logs)
- [ ] Feature flag `treatment_records` blokuje dostęp do wszystkich nowych endpointów gdy wyłączony

### Bezpieczeństwo (wszystkie must-pass)
- [ ] sensitive_health templates nie są dostępne dla employee przez API
- [ ] Zdjęcia w Supabase Storage są w private bucket (nie public URL)
- [ ] Audit log nie ma polityki RLS pozwalającej na DELETE
- [ ] Żaden nowy endpoint bez `getAuthContext()` guard

### Jakość
- [ ] Unit testy dla: consent gate, data_category filtering, treatment record CRUD, protocol assignment
- [ ] `npx playwright test` — istniejące e2e nie regresuję

## 7. Ryzyka v2.0

| Ryzyko | Prawdopodobieństwo | Wpływ | Mitigacja |
|--------|-------------------|-------|-----------|
| L2-D (multi-session plans) rozrasta się poza sprint | Wysokie | Opóźnienie | Potnij: najpierw schema+API, UI w osobnym sprincie |
| Builtin templates plik (~264k linii) blokuje TS/build | Średnie | Bloker techniczny | Najwcześniejszy sygnał = `npx tsc` timeout; fix: split na chunks lub lazy import |
| Photo storage koszty Supabase | Niskie | Koszt operacyjny | Limit per salon: 500MB w Professional, 5GB w Business |
| Compliance review kart zabiegowych: część `sensitive_health` nie przejdzie | Pewne | Mniej kart w v2.0 | Akceptowalne — lepiej mniej zatwierdzonych kart niż za dużo bez review |
| `getAuthContext()` missing w nowych endpointach | Niskie (bo wzorzec istnieje) | Security | Codex review po każdym sprincie API |
